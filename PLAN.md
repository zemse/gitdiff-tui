# gitdiff — TUI diff reviewer for local git

A Rust ratatui TUI that lets you review local git changes like a GitHub PR and leave comments that get written to a local markdown file. The markdown is structured so an AI agent (Claude Code) can read it, address the comments, and resolve the review.

## Goal

- Inspect diffs *without* pushing/opening a PR
- Same review ergonomics as GitHub: file list, hunks, per-line comments, suggested changes
- Output is `REVIEW.md` at repo root — a plain-text artifact an agent (or another human) can act on

## Diff source — auto-detected

Priority order (first non-empty wins):

1. **Working changes**: `git diff` (unstaged) + `git diff --cached` (staged) shown as one unified diff. Hunks tagged with `staged` / `unstaged` / `both`.
2. **Branch-vs-upstream**: if tree is clean, use `git diff <merge-base>..HEAD` against:
   - `@{upstream}` if set,
   - else `origin/main` or `origin/master`,
   - else fallback `main`/`master` locally.
3. **Explicit override**: `gitdiff <base>..<head>` CLI arg.

Show the resolved base/head in the header bar so the user knows what they're reviewing.

## Feature scope

Based on GitHub PR diff viewer features research. Marked: `[M0]` MVP, `[M1]` near, `[M2]` later.

### Diff display
- [M0] Unified diff with old/new line-number gutters
- [M0] Color: green add, red delete, neutral context, hunk header separator
- [M0] File header with status (A/M/D/R), +/- counts
- [M0] Per-file collapse/expand
- [M1] Split (side-by-side) view, toggle with `d`
- [M1] Intra-line word-level highlight
- [M1] Syntax highlighting via `syntect`
- [M2] Image / binary placeholders
- [M2] Expand context (`+`/`-` to load N more lines around hunk)
- [M2] Whitespace toggle, tab-width, wrap toggle

### Navigation
- [M0] `j`/`k` line scroll, `ctrl-d`/`ctrl-u` half-page
- [M0] `]`/`[` next/prev file, `}`/`{` next/prev hunk
- [M0] `?` shortcuts overlay
- [M0] `q` quit, `:w` save review
- [M1] File tree side pane, toggle with `e`
- [M1] `/` search within diff, `n`/`N` next/prev match
- [M1] `t` fuzzy file picker
- [M1] `gg`/`G` top/bottom
- [M2] Jump next/prev comment

### Commenting
- [M0] `c` on a line opens composer; write markdown; `ctrl-s` save (draft), `esc` cancel
- [M0] `V` to range-select then `c` for multi-line comment
- [M0] Draft list pane (`R` opens), `S` submits all drafts to `REVIEW.md`
- [M0] `x` delete draft on a line
- [M1] Edit existing draft (`e` over a commented line)
- [M1] Suggested-change blocks (` ```suggestion `) recognized + rendered as mini-diff in the composer preview
- [M1] Reply/thread under existing comments in `REVIEW.md` (re-loaded on open)
- [M2] Resolve / unresolve threads (move to "Resolved" section)
- [M2] Reactions (just text, e.g. `+1`)

### Review metadata
- [M0] Header: base..head, files-changed count, total +/-, draft count
- [M0] Footer: keybind hints, current file path:line
- [M1] Per-file viewed checkbox (`v`) — persists in `.gitdiff/viewed.json` under repo root, keyed by file path + content hash
- [M1] Stats bar per file (+/- numbers with mini bar)
- [M2] Filter by commit (when reviewing branch diff)

Design choices:
- One section per file, one subsection per comment thread
- The exact diff snippet is embedded so an agent can find the spot even if line numbers drift
- `· open` / `· resolved` status line so an agent knows which to address
- A ` ```suggestion ` block (GitHub-style) means "apply this verbatim" — the agent should replace the marked lines with it
- Re-running gitdiff on a repo with an existing `REVIEW.md` loads prior threads back in for re-review

### State files at repo root

```
REVIEW.md          # the human/agent-facing review artifact
.gitdiff/
  viewed.json      # per-file viewed checkbox state (M1)
  drafts.json      # in-progress comments not yet finalized (M0)
```

Add `.gitdiff/` to suggested `.gitignore` block on first run.

## Architecture

```
src/
  main.rs       # CLI parse, terminal setup/teardown, event loop
  git.rs        # shell out to `git` for status, diff, branch, merge-base
  diff.rs       # parse unified diff into Vec<FileDiff{ Hunk{ Line } }>
  app.rs        # AppState: files, cursor, mode, drafts, viewed
  ui/
    mod.rs      # layout
    diff_view.rs    # the diff pane renderer
    file_list.rs    # left sidebar (M1)
    composer.rs     # comment input modal (uses tui-textarea)
    help.rs         # ? overlay
  review.rs     # serialize/deserialize REVIEW.md and drafts.json
  keymap.rs     # keybinding table → action enum
```

Dependencies (minimal):
- `ratatui` — TUI rendering
- `crossterm` — terminal backend + input
- `anyhow` — error type
- `serde` + `serde_json` — drafts/viewed state
- `tui-textarea` — multi-line comment input
- `chrono` — timestamps
- `syntect` — syntax highlighting (M1, defer)

Shell out to `git` (no `git2` crate) — keeps the dep tree small, mirrors what the user types anyway.

## Milestones

- **M0 — Walking skeleton (this session)**: load diff (working + branch fallback), render unified view with colors, navigate j/k/]/[, add single-line comment via `c`, submit drafts to `REVIEW.md` with `S`, quit with `q`. No file tree, no syntax highlighting, no split view.
- **M1**: file tree sidebar, viewed checkbox, syntax highlighting, range comments, search, fuzzy file picker, split view, edit drafts.
- **M2**: outdated comment handling on re-review, resolve threads, expand-context, suggestion application, whitespace/tab toggles.

## Open questions (not blocking M0)

- Should `REVIEW.md` be committed or gitignored? Default: leave decision to user, but suggest gitignore on first run.
- Multi-author scenarios: for now assume single reviewer. Later: derive author from `git config user.name`.
- Conflict between an existing `REVIEW.md` and fresh drafts: merge by L-anchor, prompt on collision.
